## Problem Statement

ECMWF's operational weather forecasting pipelines rely primarily on GRIB (1 and 2) for encoding and transmitting meteorological data. GRIB is an international standard developed by WMO, which means vocabulary evolution (new parameter types, metadata concepts, level types) requires international negotiation — a process that takes months. This creates concrete delays in implementation of new processed products that don't match existing WMO descriptions.

Additionally, GRIB is fundamentally limited to 1-Tensors (single fields). Multi-dimensional data like sea wave spectra — which should be represented as a 3-tensor (spatial lat-lon, parameter, frequency) — must be flattened into sequences of GRIB messages with ad-hoc conventions, creating persistent technical debt.

A developer retreat and brainstorming session concluded that ECMWF would move faster with a flexible internal format that combines NetCDF's metadata flexibility with GRIB's performance characteristics and message-format approach. GRIB remains the right choice for external clients and archival (strict international format). Tensogram is for the internal operational pipeline where speed of vocabulary evolution and N-dimensional tensor support matter.

## Demand Evidence

- **Concrete delays:** Implementation of new processed products blocked by WMO vocabulary evolution wait times and internal mapping updates.
- **Technical debt:** Developers work around GRIB's limitations, creating accumulating workarounds that become permanent fixtures in the codebase.
- **Developer consensus:** A retreat and brainstorming session among developers concluded that a flexible internal format is needed.
- **Structural limitation:** GRIB cannot express N-dimensional data natively. Sea wave spectra (a 3-tensor of spatial, parameter, frequency dimensions) are currently split across multiple GRIB messages with ad-hoc conventions.
- **ML model outputs:** New AI/ML weather models produce N-dimensional tensors that don't map cleanly to GRIB's 1-tensor model.

## Status Quo

- **GRIB 1 & 2:** Primary workhorse format for all operational data. Good tooling ecosystem (ecCodes). Users value the metadata and tool availability. But vocabulary is WMO-regulated and structurally limited to 1-tensors.
- **NetCDF / HDF5:** Minimal internal use, mostly for static data. Users appreciate metadata flexibility.
- **BUFR / ODB:** Separate column formats for observations. Potential future unification target (time-series observations as 1-tensors in Tensogram), but explicitly deferred.

## Target User & Narrowest Wedge

**Target users:** ECMWF developers working on operational weather forecasting pipelines — the teams producing, processing, and consuming gridded forecast data internally.

**Narrowest wedge (in priority order):**
1. **Existing products with structural limitations** — data like sea wave spectra that GRIB flattens into multiple messages with ad-hoc conventions. Encoding these as proper N-tensors eliminates structural tech debt. This is the sharpest first wedge: existing data, existing pain, existing customers. (Reprioritized based on cross-model review — ML outputs follow trivially once N-tensor support works.)
2. **ML model outputs** — N-dimensional tensors from AI weather models that cannot be expressed in GRIB.
3. **New processed products** — data products with parameters that lack WMO vocabulary definitions, currently blocked or delayed.

**Deferred:** Observation data unification (BUFR/ODB replacement) — possible future work if Tensogram proves out.

## Constraints

- Must be production-ready for ECMWF operational pipelines
- Network-transmissible message format, not a file format (but appendable to files)
- Limited dependencies — lightweight enough to embed in pipeline components
- Must support existing encoding approaches (simple_packing, szip/libaec) for compatibility with operational data characteristics
- Vocabulary remains externally managed (ECMWF MARS language) — library is vocabulary-agnostic
- Must support forward evolution via version metadata entry

## Premises

1. **CBOR for all metadata** — free-form string keys, strongly-typed values (RFC 8949). The library is vocabulary-agnostic; ECMWF's MARS language vocabulary is the application layer's concern. CBOR chosen for flexibility over fixed binary structs because messages are tens to hundreds of MiB where CBOR parsing overhead is negligible compared to payload decode. Every message includes a `version` entry for forward compatibility.

2. **Self-describing messages, externally-managed vocabulary** — each message carries enough CBOR metadata to be interpreted, but the library doesn't validate or interpret vocabulary semantics.

3. **Minimal unnecessary memory copies** — tensor payloads are encoded (lossy or lossless) and optionally compressed. Supported encodings: simple_packing (GRIB-style lossy quantization) and szip/libaec (lossless compression). More encoding formats expected as the library evolves.

4. **Message framing: `TENSOGRM` (8 bytes) ... `39277777` (8 bytes)** — mixed-pattern terminator for corruption resistance. Each message independently parseable. Multiple messages appendable to files.

5. **Hybrid architecture** — binary header index + CBOR metadata sections + binary-encoded tensor payload sections within each message.

6. **Limited dependencies** — tinycbor/libcbor (or Rust CBOR crate) + libaec. No heavy frameworks.

## Cross-Model Perspective

An independent Claude subagent reviewed the design premises cold. Key findings:

- **Steelman:** "A domain-specific wire format that decouples ECMWF's internal data flow from WMO's glacial standardization process, letting them ship new products in days instead of years while preserving the encoding tricks that make GRIB viable at planetary scale."

- **Key insight on wedge:** The subagent argued sea wave spectra is a sharper first wedge than ML outputs because it's an *existing product* with *existing pain* — ECMWF already produces the data and already suffers from flattening a 3-tensor. ML people already have safetensors/zarr. "Build for that case first, and ML outputs follow trivially."

- **Challenged premise (CBOR for all metadata):** Argued tensor descriptors (shape, dtype, encoding params) are fixed-schema and performance-critical — a fixed binary struct would allow pointer-cast access. **User defended:** at message sizes of tens to hundreds of MiB, CBOR parsing is negligible; and fixing the descriptor struct would create the same rigidity Tensogram is designed to escape when adding new encoding formats.

- **48-hour prototype:** C99 + tinycbor + libaec, four core functions, round-trip a 0.1-degree global temperature field with simple_packing at 16 bits.

## Approaches Considered

### Approach A: Pure C99 Library (Minimal Viable)
C99 library with tinycbor + libaec. Four core functions. Makefile build. Maximum portability, minimum overhead. But tedious buffer management in C99 and no native higher-level language support. Completeness: 7/10.

### Approach B: C++ Core with C API (Ideal Architecture)
C++17 core with RAII buffer management, stable C API for FFI. CMake build. Safer internals, easy extensibility via strategy pattern, integrates with ECMWF build infrastructure. But adds C++ build complexity. Completeness: 9/10.

### Approach C: Rust Core with Multi-Language APIs (Creative/Lateral)
Rust implementation with cbor crate + libaec bindings. Exposes Rust native API, C FFI (for C++ interop), and Python API (via PyO3 or similar). Memory safety guaranteed by the compiler. Excellent tooling (cargo, built-in testing/benchmarking). Higher adoption friction at ECMWF but eliminates entire class of memory safety bugs. Completeness: 8/10.

## Recommended Approach

**Approach C: Rust Core with C++, Rust, and Python APIs.**

User selected this approach, citing the value of memory safety for a binary format library running in production pipelines. The multi-language API surface (Rust native + C FFI for C++ interop + Python via PyO3) covers the full ECMWF development stack.

**Rust adoption risk mitigation:** If Rust adoption proves infeasible at ECMWF (toolchain, developer availability, organizational friction), the design is language-agnostic — the wire format and CBOR metadata schema can be re-implemented in C++17 with the same C API surface. The Rust prototype would still serve as a reference implementation and test oracle. Cross-language golden test files ensure any reimplementation produces identical output.

### Implementation Architecture

**Core library (Rust):**
- `tensogram-core` crate — message encode/decode, CBOR metadata handling, framing, buffer + file API
- `tensogram-encodings` crate — simple_packing, szip/libaec integration, extensible for new formats
- `tensogram-ffi` crate — C-compatible FFI surface using `cbindgen`
- `tensogram-python` crate — Python bindings via PyO3/maturin
- `tensogram-cli` crate — CLI binary (`tensogram` command with subcommands)

**Core API — Buffer interface (primary):**

The buffer interface operates on in-memory byte slices. This is the most common usage path for pipeline components that receive data over the network or from shared memory.

- `encode(metadata, data_objects) → Vec<u8>` — encode a complete message to a byte buffer
- `decode(bytes) → (metadata, data_objects)` — decode all data objects from a message buffer
- `decode_metadata(bytes) → metadata` — decode only metadata, skip payload (uses binary header object offsets)
- `decode_object(bytes, index) → data_object` — decode a single data object by index (O(1) random access via binary header object_offsets[])
- `decode_range(bytes, object_index, ranges) → partial_data` — decode a partial sample range from a data object. Uses szip_block_offsets for compressed data (GribJump-style), or direct byte offset for uncompressed data. `ranges` is a list of (offset, count) pairs for multi-range extraction with bucket merging.
- `scan(bytes) → Vec<message_offset>` — scan a multi-message buffer, return offset/length of each message (using TENSOGRM markers)

**Core API — File interface (convenience):**

The file interface wraps the buffer interface for working with files that contain one or more Tensogram messages. Handles file I/O, multi-message iteration, and streaming.

- `File::open(path) → TensogramFile` — open a file containing Tensogram messages
- `File::create(path) → TensogramFile` — create a new file for writing messages
- `file.messages() → Iterator<Message>` — iterate over messages in a file (lazy, scans for TENSOGRM markers)
- `file.append(metadata, data_objects)` — encode and append a message to the file
- `file.message_count() → usize` — count messages without decoding them
- `file.read_message(index) → Message` — random access to a specific message by index (requires initial scan)

**CLI tool — `tensogram`:**

A single binary with subcommands, inspired by ecCodes' grib_* tools. Eases adoption by ECMWF developers familiar with the ecCodes workflow. All subcommands process messages one at a time from input file(s).

Common options across subcommands:
- Input files as trailing positional arguments: `tensogram <verb> [options] file1.tgm file2.tgm`
- `-w key=value` — where-clause filter, process only messages matching criteria. Supports `!=` (negation), `/` for OR within values (e.g., `-w mars.param=2t/10u`), namespaced keys via dot notation.
- `-p key1,key2,...` — select which metadata keys to display/extract
- `-j` — JSON output format

| Subcommand | Description |
|-----------|-------------|
| `tensogram info <file>` | Show file-level summary: message count, total size, format version. Quick health check. |
| `tensogram ls [options] <files>` | List metadata from messages in tabular format. `-p` selects keys, `-w` filters. Tolerant of missing keys (prints `N/A`). |
| `tensogram dump [options] <files>` | Full dump of message contents: all metadata keys + optionally data values (`-d`). Supports `-j` for JSON output, `-p` for specific keys. |
| `tensogram get [options] <files>` | Extract specific key values. Strict: errors if requested key is missing (designed for scripting). Same `-p`, `-w` options as `ls`. |
| `tensogram set -s key=val,... <input> <output>` | Modify metadata key/value pairs, write to output file. `-w` filters which messages are modified (non-matching pass through unchanged). Only modifies CBOR metadata, does not re-encode payloads. |
| `tensogram copy [options] <input> <output>` | Copy/split messages from input to output. Output filename supports `[keyName]` placeholders for automatic splitting (e.g., `output_[mars.param]_[mars.level].tgm`). `-w` filters which messages are copied. |
| `tensogram filter <rules_file> <files>` | Programmable rules engine. Rules file contains conditionals and write directives for complex message routing/transformation. (v2 — implement after core subcommands are stable.) |

**Wire format (per message):**

All fixed-width framing fields use big-endian (network) byte order.
Payload byte order is per-data-object, specified in metadata (see below).
CBOR encoding MUST use deterministic encoding (RFC 8949 Section 4.2) with canonical key ordering (lexicographic by key bytes) to ensure byte-identical output across implementations.

```
Offset    Size     Field
--------  -------  ----------------------------------------
0         8        Magic: "TENSOGRM" (ASCII, 0x54454E534F47524D)
8         8        Total length (uint64 big-endian, includes magic + terminator)
16        8        Metadata offset (uint64 big-endian, byte offset of CBOR block from message start)
24        8        Metadata length (uint64 big-endian, byte count of CBOR block)
32        8        Number of data objects N (uint64 big-endian)
40        N×8      Object offsets[0..N-1] (N × uint64 big-endian, byte offsets of each OBJS marker from message start)
40+N×8    var      CBOR metadata map (variable-length, see structure below)
var       var      Data object 0: OBJS (4B) | payload bytes | OBJE (4B)
          var      Data object 1: OBJS (4B) | payload bytes | OBJE (4B)
          ...      (one data object per entry in "payload" array)
N-8       8        Terminator: "39277777" (ASCII, 0x3339323737373737)
```

Where N = total length (last field value), not num_data_objects.
`OBJS` = 0x4F424A53 (ASCII), `OBJE` = 0x4F424A45 (ASCII).

The metadata_offset field accommodates optional padding bytes between the binary header and the CBOR block (e.g., for alignment). In the common case with no padding, metadata_offset = 40 + num_data_objects × 8.

**CBOR metadata structure:**

```
{
  "version": 1,                          // uint, required

  // --- Common metadata (shared across all data objects) ---
  // Organized in namespaces (sub-objects) for clarity
  "mars": {                              // MARS vocabulary namespace
    "class": "od",
    "type": "fc",
    "date": "20260401",
    "time": "1200"
  },
  ... (arbitrary namespaced scientific metadata) ...

  // --- Per-object metadata (entries that differ between objects) ---
  "objects": [
    {
      "type": "ntensor",                 // data object type
      "ndim": 3,                         // number of dimensions
      "shape": [1801, 3600, 30],         // dimension sizes (row-major/C order)
      "strides": [108000, 30, 1],        // stride per dimension (in elements)
                                         // flat_index = dot(index_vector, strides)
      "dtype": "float64",               // decoded logical type (not on-wire storage)
      "mars": {                          // per-object MARS keys (namespace)
        "param": "wave_spectra"
      },
      "units": "m^2 s",                 // scientific metadata
      ... (arbitrary per-object metadata) ...
    },
    {
      "type": "ntensor",
      "ndim": 2,
      "shape": [1801, 3600],
      "strides": [3600, 1],
      "dtype": "float32",
      "mars": {
        "param": "2t"
      },
      "units": "K"
    }
  ],

  // --- Payload decoding instructions (per data object) ---
  //     Pipeline: encode → filter → compress (reverse for decode)
  //     Contains everything needed to decode each object
  //     Byte offsets and lengths live in the binary header (object_offsets[])
  "payload": [
    {
      "byte_order": "big",              // byte order of this object's raw data
      "encoding": "simple_packing",      // lossy quantization to integers
      "filter": "none",                  // no filter needed (already integers)
      "compression": "szip",             // libaec lossless compression
      "reference_value": 230.5,          // encoding-specific params
      "binary_scale_factor": -2,
      "decimal_scale_factor": 0,
      "bits_per_value": 16,
      // --- Partial decoding support (GribJump-style) ---
      "szip_rsi": 128,                  // Reference Sample Interval
      "szip_block_size": 32,            // AEC block size
      "szip_flags": 0,                  // AEC encoding flags
      "szip_block_offsets": [0, 4128, 8240, ...],  // bit offsets of RSI block
                                         // boundaries in compressed stream
                                         // (computed during encoding, enables
                                         // partial range decoding without
                                         // decompressing entire bytestream)
      // --- Integrity hash (computed during encoding) ---
      "hash": {
        "type": "xxh3",                  // hash algorithm: "xxh3", "sha1", or "md5"
        "value": "a1b2c3d4e5f6..."       // hex-encoded hash of the encoded payload
      }
    },
    {
      "byte_order": "little",           // client encoded natively, no byte flip
      "encoding": "none",               // raw float32 data
      "filter": "shuffle",              // byte shuffle for better compression
      "shuffle_element_size": 4,         // 4 bytes per float32 element
      "compression": "szip",            // compress shuffled bytes as uint8
      "szip_rsi": 128,
      "szip_block_size": 32,
      "szip_flags": 0,
      "szip_block_offsets": [0, 2064, 4112, ...],
      "hash": {
        "type": "xxh3",
        "value": "f7e8d9c0b1a2..."
      }
    }
  ]
}
```

**Strided memory layout:** Each data object describes its N-Tensor via shape + strides, separating the logical tensor structure from the raw memory buffer (as in NumPy, PyTorch, ndarray). The 1D flat index for a multi-dimensional index vector `I` is: `flat_index = sum(I[k] * strides[k] for k in 0..ndim)`. This enables non-contiguous views, transposed tensors, and sliced sub-tensors to be described without copying data.

**Key design decisions:**
- **Binary header index** (total_length, metadata_offset, metadata_length, num_data_objects, object_offsets[]) precedes the CBOR block. This has fixed, deterministic size for any given N, enabling O(1) random access to any data object without CBOR parsing. It also eliminates the CBOR circular dependency: since offsets live outside CBOR, the CBOR block can be written at a fixed position without knowing data object positions in advance.
- **Metadata offset field** accommodates optional padding between binary header and CBOR block (e.g., for cache-line alignment). Decoders seek to metadata_offset rather than computing 40 + N×8.
- **Total length field** enables seeking past messages without any parsing.
- **Two-level metadata:** common entries (shared) + per-object entries (in `"objects"` array). Avoids duplicating shared metadata across objects.
- **Payload directory** (`"payload"` array) provides encoding pipeline parameters per data object. Byte offsets and payload lengths live in the binary header's object_offsets[], not CBOR.
- **Per-object byte order** — `"byte_order"` per data object allows clients to encode large payloads in native endianness without byte-flipping, while framing fields remain network byte order.
- **OBJS/OBJE markers** per data object provide corruption detection at the object level, not just message level.
- **Version entry** in every message enables forward-compatible evolution.

### Encoding Support (initial)

**Supported dtype values:**

| dtype string | Byte width | Description |
|-------------|-----------|-------------|
| `"float16"` | 2 | IEEE 754 half precision |
| `"bfloat16"` | 2 | Brain floating point (Google Brain format) |
| `"float32"` | 4 | IEEE 754 single precision |
| `"float64"` | 8 | IEEE 754 double precision |
| `"complex64"` | 8 | Two float32 (real, imaginary) |
| `"complex128"` | 16 | Two float64 (real, imaginary) |
| `"int8"` | 1 | Signed 8-bit integer |
| `"int16"` | 2 | Signed 16-bit integer |
| `"int32"` | 4 | Signed 32-bit integer |
| `"int64"` | 8 | Signed 64-bit integer |
| `"uint8"` | 1 | Unsigned 8-bit integer |
| `"uint16"` | 2 | Unsigned 16-bit integer |
| `"uint32"` | 4 | Unsigned 32-bit integer |
| `"uint64"` | 8 | Unsigned 64-bit integer |
| `"bitmask"` | 1/8 | Packed bits (8 bits per byte, MSB-first). Tensor elements are single bits; byte width for stride/offset calculations is 1/8. Padding bits in the final byte are zero-filled. |

**Note on sizes:** Data objects may exceed 4 GiB. All offsets and lengths in the binary header are uint64.

**Encoding pipeline:** Encoding, pre-processing filters, and compression are specified per data object in the `"payload"` array:
- `"encoding"` — data encoding: `"none"`, `"simple_packing"` (or future formats)
- `"filter"` — optional pre-processing: `"none"` (default), `"shuffle"` (or future filters)
- `"compression"` — optional compression applied last: `"none"` (default), `"szip"` (or future formats)

The full pipeline is: **encode → filter → compress** (and reverse for decoding). Each data object in a message can use a different pipeline. Every step in the pipeline MUST be fully described in the `"payload"` metadata entry for that object — decoders must never make assumptions about what was applied.

**Encoding details:**

1. **None** (`"encoding": "none"`) — unencoded tensor data in the logical dtype. Byte order is per-data-object, specified by the `"byte_order"` key in that object's metadata (`"big"` or `"little"`). This allows clients encoding large payloads to write native endianness directly without byte-flipping. Default convention: `"big"` (network byte order, matching GRIB) when not specified.

2. **Simple packing** (`"encoding": "simple_packing"`) — GRIB-style lossy quantization: `value = reference_value + 2^E * 10^(-D) * packed_integer`, where E = binary_scale_factor, D = decimal_scale_factor. Sign convention matches GRIB: D and E are signed integers, the formula uses -D (so positive D means more decimal precision). Supports 0 to 64 bits per value. At 0 bits, all values are equal to reference_value (constant field optimization). Payload metadata carries: `reference_value` (float64), `binary_scale_factor` (int), `decimal_scale_factor` (int), `bits_per_value` (uint, 0-64). Packed integers are stored MSB-first (matching GRIB convention), final byte zero-padded on the right. NaN inputs are rejected with EncodingError.

**Pre-processing filter details:**

1. **None** (`"filter": "none"`) — no pre-processing.

2. **Shuffle** (`"filter": "shuffle"`) — byte-level shuffle (as in HDF5/Blosc). Groups the first byte of every element together, then the second byte, etc. For a float32 tensor, this groups sign+exponent bits, then mantissa bytes, dramatically improving compression ratios for compressors that exploit byte-level correlation. Essential for compressing non-integer types with szip/libaec, which only encodes integers — after shuffle, the rearranged bytes are passed to the compressor as 8-bit unsigned integers. Payload metadata carries: `shuffle_element_size` (uint, byte width of the dtype before shuffle).

**Compression details (v1 — initial implementation):**

1. **None** (`"compression": "none"`) — no compression.

2. **szip (libaec)** (`"compression": "szip"`) — lossless compression implementing CCSDS 121.0-B-3. **Important limitation:** libaec only encodes integers natively. For non-integer types, one of two approaches must be used (fully described in metadata):
   - Lossy: apply simple_packing first to quantize to integers, then compress
   - Lossless: apply shuffle filter first (rearranges bytes of float/complex types), then compress the shuffled bytes as uint8

   **Partial range decoding (GribJump-style):** AEC compresses data in RSI blocks (Reference Sample Interval, typically 128 x block_size samples). During encoding, the library records the bit offset of each RSI block boundary in the compressed stream and stores them in the payload metadata as `szip_block_offsets`. This enables partial range decoding:

   1. Map the requested sample range to the enclosing RSI block indices
   2. Use `szip_block_offsets` to locate the compressed bytes for those blocks only
   3. Decompress only those RSI blocks (libaec has no `aec_decode_range()` — implement RSI block seeking manually, ~200-300 LOC in `partial_decode.rs`, porting GribJump's Aec.h/Ccsds.h approach)
   4. Extract the requested samples from the decompressed block(s)
   5. Apply the reverse encoding transform (e.g., simple_packing → float)

   Granularity: the minimum skip unit is one RSI block (typically 4096 samples). Required payload metadata: `szip_rsi`, `szip_block_size`, `szip_flags`, `szip_block_offsets` (array of uint64 bit offsets).

   For uncompressed data (`"compression": "none"`), partial range decoding is trivial — direct byte offset calculation from the binary header object_offsets[] entry + sample index × element size.

**Note on shuffle + partial range decode:** These do not compose. Partial range decode only works for `simple_packing+szip` and uncompressed data. For shuffled data, the byte rearrangement means the RSI block boundaries do not correspond to contiguous sample ranges in the original tensor. Document this limitation clearly; do not attempt to combine shuffle with partial range decode.

**Compression roadmap (planned, not v1):**

The encoding pipeline and metadata structure are designed to accommodate these without wire format changes — each is just a new `"compression"` or `"encoding"` string value with its own metadata parameters.

| Compression | Type | Notes |
|------------|------|-------|
| `"blosc2"` | Lossless | Multi-threaded, supports multiple internal codecs. Research: dtype support, thread safety in FFI. |
| `"zstd"` | Lossless | Excellent ratio/speed tradeoff. Widely available. Research: streaming vs block mode for large tensors. |
| `"lz4"` | Lossless | Fastest decompression. Good for real-time pipelines. |
| `"zfp"` | Lossy | Purpose-built for floating-point arrays. Research: fixed-rate vs fixed-precision modes, dimension limits. |
| `"sz3"` | Lossy | Error-bounded lossy compression for scientific data. Research: supported dtypes, error bound specification in metadata. |

**Implementation note on compression limitations:** Each compression algorithm has dtype and dimensionality constraints. When implementing new compressors, thoroughly research:
- Which dtypes are natively supported (integer-only like libaec? float-only like ZFP?)
- Whether the shuffle filter or simple_packing is needed as a pre-processing step
- Maximum dimensions and sizes supported
- All parameters needed for correct decoding

All such details MUST be recorded in the payload metadata. The decoder must be able to reconstruct the original data from metadata alone, with zero implicit assumptions.

**Integrity hashing:**

Each data object's payload entry includes a `"hash"` sub-object with a `"type"` and `"value"` (hex-encoded). Supported hash algorithms:

| Hash type | Output size | Notes |
|----------|------------|-------|
| `"xxh3"` | 64-bit | Default. Non-cryptographic, extremely fast (~30 GB/s). Best for integrity verification. |
| `"sha1"` | 160-bit | Cryptographic (legacy). Widely available. |
| `"md5"` | 128-bit | Cryptographic (legacy). For compatibility with existing ECMWF infrastructure. |

The hash is computed over the final encoded payload bytes (after the full encode → filter → compress pipeline).

**Encoding behavior:** Hash computation is on by default (xxh3). The encoder accepts an option to disable hashing or select a different algorithm. Since quantization and compression are typically slower than hashing, the performance impact is minimal.

**Decoding behavior:** Hash verification is off by default. The decoder accepts an option to enable verification. This design reflects ECMWF's operational reality: internal transmissions typically use error correction at the transport layer, so double-checking every payload is unnecessary overhead. Verification is available for scenarios where data traverses untrusted paths or for archival validation.

**Encoding capability note:** Producers must not assume receivers support encodings, filters, or compressions added after the initial release. These are capability concerns, not version concerns. Out-of-band capability negotiation (or sticking to baseline encodings) is the producer's responsibility.

### Testing Strategy

- **Round-trip correctness:** Encode → decode → verify bit-exact (for lossless) or within quantization tolerance (for simple_packing)
- **Real data tests:** Use actual ECMWF data shapes (0.1-degree global fields: 1801x3600, ensemble tensors, wave spectra 3-tensors)
- **Multi-object tests:** Messages containing multiple data objects with different encodings, dtypes, and byte orders. Verify random access by index via binary header object_offsets[].
- **Strided layout tests:** Non-contiguous strides, transposed views, sliced sub-tensors — verify correct flat-index computation
- **Multi-message tests:** Append multiple messages to buffer, verify independent parsing
- **Corruption tests:** Verify framing markers detect truncated/corrupted messages
- **Cross-language tests:** Golden binary test files checked into the repo. Each language's test suite decodes the same golden files AND encodes from the same inputs, verifying byte-identical output. CI matrix runs all three language test suites on every commit.

### Error Handling & Recovery

- **Framing validation:** Decoder verifies `TENSOGRM` magic at expected offset. If missing, scanner advances byte-by-byte to find next `TENSOGRM` marker (skip-to-next-marker recovery for multi-message files).
- **Corrupted messages:** If total_length is inconsistent with terminator position, or terminator `39277777` is missing/wrong, the message is rejected. Decoder skips to next magic marker and continues.
- **Partial reads:** A message without a terminator is never valid. Partial messages at end-of-stream are reported as truncated, not silently dropped.
- **Unknown CBOR keys:** Decoders MUST ignore unknown CBOR keys (forward compatibility). Unknown encoding types are rejected with a clear error.
- **Data object validation:** Each data object's `OBJS`/`OBJE` markers are verified. If a data object is corrupted, that object is rejected but other objects in the same message may still be accessible via binary header object_offsets[].
- **Validation:** objects.len == payload.len must hold; if not, the message is rejected with MetadataError.
- **API error surface:** All functions return Result types (Rust), error codes (C FFI), and raise exceptions (Python) with structured error categories: FramingError, MetadataError, EncodingError, CompressionError, ObjectError.

### Version Compatibility Rules

- Version is a single unsigned integer starting at 1.
- **Minor evolution (same version):** Adding new CBOR metadata keys or new encoding types does NOT bump the version. Decoders ignore unknown keys and reject unknown encodings gracefully.
- **Version bump:** Only required when the wire format structure changes (field positions, framing semantics, CBOR structure changes). Decoders MUST reject messages with an unrecognized version.
- **No backward compatibility obligation across major version bumps.** The version field exists to fail fast, not to maintain legacy decode paths.

## Open Questions

1. **Maximum tensor dimensions?** Is 8 sufficient, or should ndim be unbounded? (Affects shape representation in CBOR — unbounded is natural for CBOR arrays.)
2. **Chunk/streaming support?** Should a single message support chunked payloads for very large tensors (>1 GiB)?
3. **CBOR tag usage?** Should Tensogram define custom CBOR tags for common types (datetime, geo-coordinates) or keep everything as standard CBOR types?
4. **libaec Rust bindings:** Use existing `libaec-sys` crate or vendor the C library and write custom bindings? (Investigate crate viability before coding starts — if stale, vendor libaec and write minimal FFI bindings.)
5. **ciborium canonical encoding:** Verify that ciborium's serialization produces RFC 8949 Section 4.2 deterministic output (canonical key ordering). If not, wrap with a BTreeMap-based serializer that sorts keys lexicographically before encoding.

**Resolved in this design:**
- **Byte order:** Framing fields are big-endian (network byte order). Payload byte order is per-data-object via `"byte_order"` metadata key, allowing native-endian encoding for performance.
- **Message-level checksum:** No checksum in v1. Framing markers + transport-layer checksums are sufficient. Can be added as an optional CBOR metadata key in future.
- **Dimension ordering:** Row-major (C order), matching Rust/Python/C conventions. Specified in the `shape` array.
- **dtype semantics:** `dtype` is the decoded logical type, not the on-wire storage type.
- **Object offsets location:** Object byte offsets live in the binary header (object_offsets[]), not in the CBOR objects[] entries. This eliminates the CBOR circular dependency during encoding.
- **Zero-object messages:** Valid. A message with N=0 has an empty binary header index and no payload sections. Useful for pipeline signaling via metadata-only messages.

## Success Criteria

1. **Correctness:** Round-trip encoding/decoding with bit-exact results (lossless) or verified quantization tolerance (lossy) for all supported data types and dimensions.
2. **Flexibility demonstrated:** Encode a sea wave spectrum as a proper 3-tensor with scientific metadata, encode an ML model output as an N-tensor, and encode a traditional 2D field — all using the same library with no code changes, only different metadata.
3. **Cross-language interop:** A message encoded in Rust can be decoded in Python and C++, and vice versa.
4. **Production readiness:** No memory safety issues (guaranteed by Rust), comprehensive test suite, documented API.
5. **Performance baseline:** Encode/decode throughput measured and documented for reference data sizes. No specific performance target for v1 — correctness and flexibility are the primary metrics. Performance optimization is a follow-up concern once the format is proven correct.
