# Decision: Bidirectional Remote Scan — Stay Opt-In

**Status**: rejected (defaults remain forward-only across Rust, Python, and TypeScript)
**Date**: 2026
**Mechanism**: data-driven — no flip applied

## Context

The bidirectional remote-scan walker landed across Rust, Python, and
TypeScript through prior work.  It is exposed via
`RemoteScanOptions { bidirectional: true }` in Rust, the
`bidirectional=True` keyword in Python's four file-open entry points,
and `FromUrlOptions.bidirectional: true` in TypeScript.

The walker pairs forward preamble fetches with backward postamble
fetches in the same paired round, with the design intent of roughly
halving the number of `GET` requests on full-scan and tail-access
patterns against header-indexed `.tgm` files.

This artifact records whether to flip those defaults to `true` so
existing readers gain the speedup automatically when the library
upgrades.

## Criterion

Verbatim from `plans/TODO.md` → "remote 8 — benchmarks, docs,
default-flip decision":

> bidirectional ≤ forward-only on request count AND bytes fetched at
> every tier × every scenario, across Rust + Python + TS.  Tiny `+1`
> exceptions on N=1 allowed only if wall-clock clearly wins and the
> extra request is justified.

Wall-clock is informational only; the gate is the two deterministic
metrics.

## Methodology

Three benchmark harnesses, one per language, share a fixture roster
and a `cold_open_plus_operation_plus_close` sample contract:

- **Fixtures** (header-indexed N ∈ {1, 10, 100, 1000}, footer-indexed
  N ∈ {1, 10, 100, 1000}, streaming-tail N=10): generated by
  `tests/remote-parity/tools/gen_fixtures.py`, served from the same
  bytes by every harness.
- **Scenarios** (per fixture × tier): `message_count`,
  `read_message(0)`, `read_message(N-1)`, `read_message(N/2)`, `iter`
  (`message_count` followed by `read_message(i)` for every
  `i ∈ [0, N)`).
- **Walkers**: `forward-only` (`bidirectional=false`) and
  `bidirectional` (`bidirectional=true`).

Each cell is a fresh `open_remote → operation → handle dropped`
cycle against an instrumented mock HTTP server.  The handle is not
reused across cells, so all probe + discovery + payload traffic is
attributed to the cell that issued it.  ("Close" in Python is the
context manager's `__exit__` and in TypeScript a `#closed = true`
flag — neither releases caches; the cold-handle invariant is
established by creating a fresh `TensogramFile` per cell, not by
clearing state on the existing one.)  The mock server records every
request with method, path, range header, status, and response body
size; the harness extracts `total_requests`, `range_get_requests`,
`head_requests`, and `response_body_bytes` per cell.

Per-language sidecars:

- Rust — `target/remote-scan-bench/rust.ndjson` from
  `cargo run --release -p tensogram-benchmarks --bin remote-scan-metrics`.
- Python — `target/remote-scan-bench/python.ndjson` from
  `python rust/benchmarks/python/bench_remote_scan.py`.
- TypeScript — `target/remote-scan-bench/typescript.ndjson` from
  `npm --prefix typescript run bench:remote-metrics`.

All three NDJSON files share an identical schema, one record per
(language × fixture × tier × scenario × walker) cell, and one record
emitted per single deterministic run (no Criterion-style iteration
smearing of the request counters).

## Aggregate verdict

| Language | EQUAL/PASS cells | FAIL cells | Total |
|----------|------------------|------------|-------|
| Rust | 17 | 28 | 45 |
| Python | 17 | 28 | 45 |
| TypeScript | 0 | 45 | 45 |

**Verdict: FAIL** in every language.

The `EQUAL/PASS` cells in Rust and Python are exactly the cells where
the two walkers are observationally identical: N=1 (no backward to
do) and `read_message(0)` (target is the first message; both walkers
walk forward exactly once).  In every other cell, bidirectional
strictly fetches **more** bytes and at least as many requests as
forward-only.  TypeScript shows no equal cells at all because its
walker issues an additional eager footer fetch per backward-discovered
message that the gate later discards (the bytes are wasted, the
request is not).

The criterion permits a `+1` GET exception on N=1 if wall-clock
clearly wins.  No exception applies: the actual N=1 cells are
byte-equal already.

## Headline cells (full-file scan)

For the `iter` scenario at `N=100` and `N=1000`, header-indexed:

| Lang | N | walker | total_requests | response_body_bytes | wall_ms |
|------|---|--------|---------------:|--------------------:|--------:|
| rust | 100 | forward-only | 201 | 61 600 | 215 |
| rust | 100 | bidirectional | 202 | **1 540 424** | 216 |
| rust | 1000 | forward-only | 2 001 | 616 000 | 2 110 |
| rust | 1000 | bidirectional | 2 002 | **148 604 024** | 2 114 |
| python | 100 | forward-only | 201 | 61 600 | 68 |
| python | 100 | bidirectional | 202 | **1 540 424** | 70 |
| python | 1000 | forward-only | 2 001 | 616 000 | 686 |
| python | 1000 | bidirectional | 2 002 | **148 604 024** | 747 |
| typescript | 100 | forward-only | 201 | 61 600 | 120 |
| typescript | 100 | bidirectional | 251 | 62 800 | 156 |
| typescript | 1000 | forward-only | 2 001 | 616 000 | 1 096 |
| typescript | 1000 | bidirectional | 2 501 | 628 000 | 1 341 |

Two structurally distinct failures coexist:

- **Rust + Python**: bidirectional fetches roughly 25× to 240× the
  bytes of forward-only.  This is `object_store::get_ranges`'s
  coalescer (default merge gap 1 MiB) collapsing paired forward +
  backward fetches into one contiguous Range GET whose body spans
  the file region between the two cursors.  For the `thousand-msg.tgm`
  fixture (~592 KB), bidirectional `message_count` fetches ~148 MB
  — roughly 250 paired-round fetches each pulling back ~600 KB
  instead of two 24-byte preamble reads.  The walker is correct;
  the transport's coalescer is the cost.
- **TypeScript**: `Promise.allSettled` issues each Range request
  separately, so bytes stay close to forward-only (+2% at N=1000).
  Requests, however, scale at +50% because each backward-discovered
  message triggers an eager footer fetch that the gate inside
  `tryApplyEagerFooter` later discards (no `FOOTER_INDEX` flag on
  header-indexed messages).  The discarded bytes cost a few hundred
  per message; the wasted GET cost is structural.

## Streaming-tail cell evaluation

| Lang | scenario | fwd req | bid req | fwd bytes | bid bytes | verdict |
|------|----------|--------:|--------:|----------:|----------:|---------|
| rust | iter | 22 | 22 | 6 232 | 11 600 | FAIL (bytes) |
| rust | read_message(0) | 3 | 3 | 616 | 616 | EQUAL |
| python | iter | 22 | 22 | 6 232 | 11 600 | FAIL (bytes) |
| typescript | iter | 12 | 13 | 6 224 | 6 248 | FAIL (request) |

All three walkers correctly fall back to forward when they encounter
the streaming-mode tail message — the request count stays
within `±1` of forward.  Rust and Python pay an extra coalesced
fetch in the discovery phase; TypeScript pays the eager footer fetch.
Performance-neutral within `±1` request, as required by the streaming
fallback contract; not a candidate for default flip on its own.

## Decision

**Defaults remain `false` across all three languages.**

The criterion as written admits no flip given the current numbers.
Three latent bugs were already addressed in support of this decision:

- `python/bindings/src/lib.rs:scan_opts_for` previously collapsed
  `bidirectional=False` to `None`, which the Rust layer reroutes
  through `RemoteScanOptions::default()`.  Today's default is
  forward-only so opt-out worked accidentally; any later flip would
  silently override `bidirectional=False`.  Fixed to always emit
  explicit `Some(RemoteScanOptions { bidirectional })`.
- `tests/remote-parity/drivers/rust_driver` had the same
  `.then_some(...)` chain.  Fixed to always pass explicit options.
- `typescript/src/file.ts` validated `concurrency: 1` against the
  literal `bidirectional === true`, which would let `{concurrency: 1}`
  bypass the guard once the default flipped.  Lifted to use a
  pre-computed `effectiveBidirectional` so the validation reflects
  the operative value.

These fixes ship regardless of the flip outcome — they harden the
opt-out contract.

## What would have to change for a future flip

The walker is shipping correctly today; the cost lives in the
transport layer:

- **Rust + Python**: tune or bypass `object_store`'s range coalescer
  for paired discovery fetches, or stop using paired ranges for
  preamble walks (use suffix-range backward postamble reads only,
  letting the coalescer's heuristic match the access pattern).
- **TypeScript**: skip the eager footer fetch when the backward-
  discovered message's preamble flags clearly say header-indexed.
  The gate currently runs after the fetch; pushing it before the
  fetch (synchronous flag check on the just-fetched preamble) saves
  one round trip per backward-discovered message.

Either change is out of scope for this work and can be revisited
under the deferred *remote 8 — adaptive direction* sub-task.
