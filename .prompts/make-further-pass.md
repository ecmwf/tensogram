# Further Pass Review

Perform a quality review pass over the codebase or the specified area.

**This is pass number $ARGUMENTS (default: 2 if not specified).**

Track the pass number and increase strictness with each successive pass:

## Pass 1-2 (Foundation)
- Simplification opportunities — reduce complexity, remove redundancy
- Naming quality — variables, functions, types, modules
- Comments and doc quality — accurate, helpful, not redundant
- Identify duplicate sources of truth introduced or touched by this change
  (the same value or generated content kept in two places without a guard)
- Running required formatter/lint/tests

## Pass 3-4 (Hardening)
Everything from Pass 1-2, PLUS:
- Scan for edge-cases and logical regressions
- No panics in Rust code (`unwrap()`, `expect()`, `panic!()`)
- All documentation up-to-date with changes
- Error messages are clear and actionable
- Public API surface is minimal and clean
- No dead code, no unused imports, no stale TODOs

- Generated / derived-artefact drift scan. Each hit must be inspected; the
  greps are not auto-fail rules — most matches are legitimate, but every
  match is a candidate for a hand-vendored copy that has drifted from its
  source.

  ```bash
  # Files that look like generator output:
  git ls-files -z \
    | xargs -0 grep -IlE \
        '(@generated|generated by|automatically generated|do not edit|Code generated|cbindgen|wasm-bindgen|openapi-generator|swagger-codegen|protoc|prost|bindgen|ts-rs)'

  # Likely duplicate basenames among those generated files:
  git ls-files -z \
    | xargs -0 grep -IlE '(@generated|generated by|automatically generated|do not edit|Code generated)' \
    | awk -F/ '{print $NF}' | sort | uniq -d

  # Generator invocations across build files:
  git grep -nE '(cbindgen|wasm-bindgen|protoc|prost-build|openapi-generator|swagger-codegen|bindgen|ts-rs|schemars|maturin|wasm-pack)' \
    -- Makefile '**/*.mk' '**/*.sh' '**/CMakeLists.txt' '**/*.cmake' \
       '.github/workflows/*.yml' 'package.json' '**/build.rs'
  ```

  For each generated-looking file in more than one tracked path: require
  either a single canonical consumer path or a CI regenerate-and-diff
  guard that fails on drift.

- Build-wrapper freshness scan. Same "must inspect" rule.

  ```bash
  # CMake wrappers around nested build tools:
  git grep -n -C4 -E 'add_custom_command|OUTPUT|COMMAND +(cargo|npm|npx|yarn|pnpm|make|maturin|wasm-pack|docker|bazel|gradle|mvn)' \
    -- '**/CMakeLists.txt' '**/*.cmake'

  # Shell / Make / CI wrappers that gate a nested build tool on artefact existence:
  git grep -nE '(\[\[? -[ef] |test -[ef] |if .* -[ef] ).*(target/|build/|dist/|\.a|\.so|\.dylib|\.whl|node_modules|pkg/)' \
    -- Makefile '**/*.mk' '**/*.sh' '.github/workflows/*.yml' 'package.json'
  ```

  Suspicious: the wrapper checks for an artefact and skips a nested build
  tool. The inner tool has its own incremental cache; outer-tool mtime
  caching can serve a days-old build to today's consumers.

  Acceptable: `add_custom_command(OUTPUT ...)` when one input file produces
  one generated output, all inputs are listed in `DEPENDS`, byproducts are
  declared, and there is no nested incremental build tool hidden behind
  the command.

## Pass 5+ (Polish)
Everything from Pass 3-4, PLUS with ZERO TOLERANCE:
- Every public function has doc comments
- Every error path is tested
- Every edge case has a test
- Code reads like well-written prose — a newcomer could understand it
- Performance: no unnecessary allocations, clones, or copies
- Consistency: similar patterns handled the same way everywhere
- Cross-language parity: Rust, Python, C FFI, C++ all expose equivalent functionality

- Clean-room rebuild for changed build graphs. If the change touches
  generators, build wrappers, CI, packaging, or install layout, delete the
  relevant build outputs (`cargo clean`, `rm -rf build/`, `rm -rf dist/`,
  `npm ci`, etc., as appropriate) and rebuild + retest from scratch. The
  command must exercise the affected path, not just the default aggregate
  target. Stale-cache regressions only show up on a fresh build, and they
  are silent until they bite a downstream consumer.

## Process

1. State which pass number this is and what strictness level applies
2. Scan the codebase (or specified files/area)
3. List all findings grouped by category
4. Fix each finding, verifying the fix compiles and tests pass
5. Run all formatters and linters: `cargo fmt`, `cargo clippy`, `ruff check`, `ruff format`
6. Run all tests: `cargo test --workspace`, `python -m pytest python/tests/ -v`
7. Summarize what was changed and what the next pass should focus on
